Repository for the **docu-upload** DevOps project — a microservices platform for document upload with semantic search powered by vector embeddings.

This repository contains four independent Spring Boot services, a React/TypeScript frontend, a complete Docker Compose environment, and a Jenkins CI/CD pipeline. The end-to-end workflow is: a user uploads a document → the document is stored in MongoDB → an event is published on RabbitMQ → the notification service creates a notification and the search service indexes the document's embedding in Qdrant → the user can query documents using natural-language semantic search.

## What this repository does

The platform supports two connected goals:

1. **Provide a secure multi-user document management system** with JWT-based authentication, file upload/download, and CRUD operations.
2. **Enable semantic search over uploaded documents** using vector embeddings generated by a locally running Ollama model and indexed in Qdrant.

The system consists of:

- **4 Spring Boot microservices** each with its own database and responsibility boundary
- **1 React + TypeScript frontend** consuming all service APIs
- **7 containerised infrastructure components** (PostgreSQL, MongoDB, RabbitMQ, Qdrant, Ollama, InfluxDB, Jenkins)
- **1 shared Maven module** (`common`) providing JWT utilities, security filters, and AOP-based performance tracing

```
        │                          └── MongoDB    (db_documents)
        ├── notification-service :8083  PostgreSQL (db_notifications)
        │       ▲ consumes RabbitMQ
        └── search-service :8084  ─── Qdrant (vector DB)

```

All services authenticate requests via a shared `JwtAuthFilter` and emit performance metrics to InfluxDB.

### Key directories

- `common/` — shared Maven dependency consumed by all four services; contains JWT logic and the AOP performance-monitoring aspect.
- `user-service/` — exposes `/api/auth/login`, `/api/auth/register`, and `/api/users/me`; backed by PostgreSQL.
- `document-service/` — exposes `/api/documents` (CRUD + file upload); stores metadata in PostgreSQL and content in MongoDB; publishes `DocumentEvent` messages to RabbitMQ.
- `notification-service/` — consumes `DocumentEvent` from RabbitMQ and persists notifications in PostgreSQL; exposes `/api/notifications`.
- `search-service/` — consumes `DocumentEvent` from RabbitMQ, generates a 768-dimensional embedding via Ollama (`nomic-embed-text`), and upserts it into Qdrant; exposes `/api/search?q=…&limit=…`.
- `frontend/` — single-page React application with five main routes (login, dashboard, document view/edit, semantic search).
- `docker/` — per-service Dockerfiles and SQL initialisation scripts.
- `documentation/` — functional requirements, system analysis, planning notes, and testing strategies.

## Installation

### 1. Start the infrastructure

All infrastructure services are defined in `docker-compose.yml`. Start them before launching any Spring Boot service:

```bash
docker compose up -d
```

This starts PostgreSQL, MongoDB, RabbitMQ, InfluxDB, Qdrant, Ollama, and the pgAdmin interface.

### 2. Pull the embedding model into Ollama

```bash
docker exec -it ollama ollama pull nomic-embed-text
```

### 3. Build the backend

From the repository root:

```bash
mvn clean package -DskipTests -B
```

This compiles and packages `common`, `user-service`, `document-service`, `notification-service`, and `search-service` in dependency order.

### 4. Install and build the frontend

```bash
cd frontend
npm install
npm run build
```

For development with hot reload:

```bash
npm run dev
```

## Running the services

Run each Spring Boot service from its own terminal (or adjust the command with `--server.port` to taste):

## CI/CD pipeline

The `Jenkinsfile` defines three sequential stages:

1. **Build** — `mvn clean compile -B` across all modules.
2. **Test** (parallel) — backend unit tests via `mvn test -B` with JUnit XML reporting; frontend static analysis via `npm run lint`.
3. **Package** — `mvn package -DskipTests -B`; JAR artifacts are archived with fingerprinting.

The pipeline runs inside the Jenkins container exposed on port `9090`.

## Configuration notes

- **JWT secret** is shared across services via the `common` module and configured in each service's `application.properties`. Tokens expire after `86400000 ms` (24 hours).
- **InfluxDB metrics** are opt-in per service via the `INFLUXDB_ENABLED` environment variable. The `PerformanceTraceAspect` annotates methods with `@TrackPerformance` and writes timing data to InfluxDB.
- **RabbitMQ exchange/queue names** are defined in each consuming service's configuration. The document service publishes to the exchange consumed by both the notification and search services.
- **Qdrant collection** is created on first indexing with a vector size of `768` to match the `nomic-embed-text` output dimension.
